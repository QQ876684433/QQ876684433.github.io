软件架构

架构师的工作：沟通、实现技术的评估、工程管理要有经验

架构来源

  • NFR：主要来源，非功能需求

  • 进一步划分：ASRs（识别出来的重要需求），质量需求（即功能性需求的实现质量）

  • stakeholder：对架构有不同的考虑

    • 用户：外部可观测的质量属性
    • 开发人员：内部质量属性（不可被观测到的）

  • organization：共同的商业目标

  • 技术环境：制约当前系统、兼容现有系统

架构视图（4+1）

四个视图描述的都是不同的方面

  • 逻辑视图：静态的结构问题

    描述组成和关系

  • 过程视图

    运行起来时各个部分相互之间的关系、行为

  • 物理视图

    软件系统和外界的某种联系，运行到特定的环境上、和外界环境的交互

  • 开发视图

    怎么开发出来的，给开发人员看的；不同的人、团队分工、协作（静态分工、开发过程、顺序、集成）

    时间、交付日期、资源

  • use case scenarios

    将四个视图结合起来；高优先级、低优先级场景

架构相关的活动

  • 设计
  • 实现
  • 维护
  • 演化

架构相关的知识

架构过程

  • 只关心设计阶段的活动
    • 识别ASR

  • 软件需求
