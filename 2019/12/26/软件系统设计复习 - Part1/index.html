软件架构

架构师的工作：沟通、实现技术的评估、工程管理要有经验

架构来源

  • NFR：主要来源，非功能需求

  • 进一步划分：ASRs（识别出来的重要需求），质量需求（即功能性需求的实现质量）

  • stakeholder：对架构有不同的考虑

    • 用户：外部可观测的质量属性
    • 开发人员：内部质量属性（不可被观测到的）

  • organization：共同的商业目标

  • 技术环境：制约当前系统、兼容现有系统

架构视图（4+1）

四个视图描述的都是不同的方面

  • 逻辑视图：静态的结构问题

    描述组成和关系

  • 过程视图

    运行起来时各个部分相互之间的关系、行为

  • 物理视图

    软件系统和外界的某种联系，运行到特定的环境上、和外界环境的交互

  • 开发视图

    怎么开发出来的，给开发人员看的；不同的人、团队分工、协作（静态分工、开发过程、顺序、集成）

    时间、交付日期、资源

  • use case scenarios

    将四个视图结合起来；高优先级、低优先级场景

Software Architecture in General

What is software architecture?

The software architecture of a program or computing system is the structure or structures of the system, which comprise software elements, the externally visible properties of those elements, and the relationships among them. （ SEI ）

程序或计算系统的软件体系结构是系统的一个或多个结构，其中包括软件元素，这些元素的外部可见属性以及它们之间的关系。

（一个程序或者计算系统的软件架构指的是该系统的结构，它们协调了软件中的各个元素、这些元素的外部可见属性以及它们之间的关系和联系）

KeyWord: structure, element, property, relation

The fundamental organization of a system, emboied in its components, their relationships to each other and the environment, and principles governing its design and evolution. （ IEEE ）

系统的基本组织，包括其组件，彼此之间的关系以及与环境的关系，以及控制其设计和演进的原则

（ 一个系统的基本组织，体现在它的组成部分，彼此的关系和环境，以及主导其设计和演变的原则（可维护性和可扩展性））

KeyWord: organzation, compnents, relationship, principle

What does a software architect do?

软件架构师的工作 ？

  1. Liaison 协调各方（包括客户、技术团队以及业务/需求分析师），沟通交流，包括管理和市场
  2. Software Engineering 软件工程的基本技能，软件工程的最佳实践
  3. Technology Knowledge。软件开发技术的基本知识，对技术领域有深入的理解和掌握
  4. Risk Management。 风险管理能力，包括和设计、技术选择相关的风险等

Where do architectures come from?

软件架构的来源 ?

NFRs, ASRs, Quality Requirements, Stakeholders, Organisations, Technical Environments

  1. NFRs, 非功能需求，一般包括约束和质量属性
  2. Architecture Significant Requirement 架构攸关需求，即有何没有这个需求，最后设计出来的架构可能是截然不同的
  3. Quality Requirements. 在功能需求的基础上需要满足的特征
  4. Stakeholders 系统的利益相关者，例如用户、客户、开发人员等等
  5. Organisations 开发组织
  6. Techinal Environments 技术环境，包括技术选型、兼容现有系统等等

不不仅仅局限于需求，可能是商业和技术决定的集合

Architecture Views

Modern systems are frequently too complex to grasp all at once. Instead, we restrict our attention at any one moment to one (or a small number) of the software system’s structures. To communicate meaningfully about an architecture, we must make clear which structure or structures we are discussing at the moment—which view we are taking of the architecture.

The views are used to describe the system from the viewpoint of different stakeholders, such as end-users, developers, system engineer, and project managers.

现代系统常常太复杂而无法一次掌握所有信息。 取而代之的是，我们在任何时候都将注意力集中在软件系统结构中的一个（或少数）结构上。 为了对架构进行有意义的交流，我们必须弄清当前正在讨论的是哪种结构，即我们对架构的看法。

这些视图用于从不同的利益相关者（例如最终用户，开发人员，系统工程师和项目经理）的角度描述系统。

Logical view

The logical view is concerned with the functionality that the system provides to end-users.

UML diagrams used to represent the logical view include class diagrams and state diagrams。

describes architecturally significant elements of the architecture and the relationships between them.

逻辑视图与系统提供给最终用户的功能有关。

用于表示逻辑视图的UML图包括类图和状态图。

描述了架构中具有架构重要性的元素及其之间的关系；静态的结构问题，描述组成和关系

Process view

The process view deals with the dynamic aspects of the system, explains the system processes and how they communicate, and focuses on the runtime behavior of the system. The process view addresses concurrency, distribution, integrators, performance, and scalability, etc.

UML diagrams to represent process view include the activity diagram.

Concurrency and communications elements of an architecture

流程视图处理系统的动态方面，解释系统流程及其通信方式，并着重于系统的运行时行为。 流程视图解决了并发，分布，集成器，性能和可伸缩性等问题。

表示过程视图的UML图包括活动图。

架构的并发和通信元素；运行起来时各个部分相互之间的关系、行为

Physical view

The physical view depicts the system from a system engineer’s point of view. It is concerned with the topology of software components on the physical layer as well as the physical connections between these components. This view is also known as the deployment view.

UML diagrams used to represent the physical view include the deployment diagram.

how the major processes and components are mapped on to the applications hardware

物理视图从系统工程师的角度描述了系统。 它涉及物理层上软件组件的拓扑以及这些组件之间的物理连接。 该视图也称为部署视图。

用于表示物理视图的UML图包括部署图。

主要流程和组件如何映射到应用程序硬件；软件系统和外界的某种联系，运行到特定的环境上、和外界环境的交互

Development view

The development view illustrates a system from a programmer’s perspective and is concerned with software management. This view is also known as the implementation view. It uses the UML Component diagram to describe system components.

UML Diagrams used to represent the development view include the Package diagram.

*captures the internal orgnazition of software components *

开发视图从程序员的角度说明了一个系统，并且与软件管理有关。 该视图也称为实现视图。 它使用UML组件图来描述系统组件。

用于表示开发视图的UML图包括Package图。

捕获内部保存的软件组件的组织，例如配置管理工具；怎么开发出来的，给开发人员看的；不同的人、团队分工、协作（静态分工、开发过程、顺序、集成）；时间、交付日期、资源

Architecture use cases

The description of an architecture is illustrated using a small set of use cases , or scenarios, which become a fifth view. The scenarios describe sequences of interactions between objects and between processes. They are used to identify architectural elements and to illustrate and validate the architecture design. They also serve as a starting point for tests of an architecture prototype. This view is also known as the use case view.

capture the requirements for architecture, related to more than one particular view

使用一小套用例或场景来说明对体系结构的描述，这些用例或场景成为第五种视图。 这些场景描述了对象之间以及流程之间的交互序列。 它们用于标识体系结构元素以及说明和验证体系结构设计。 它们还可以作为测试体系结构原型的起点。 该视图也称为用例视图。

架构用例：捕获架构需求；与多个特定视图相关；将四个视图结合起来；高优先级、低优先级场景

注：加粗部分是老师ppt上的描述，其他是维基百科上的描述

Quality Attributes

Software Requirements

  1. Functional requirements（功能需求）

    1. 系统必须做什么以及系统如何给利益相关者提供价值
    2. 意味着系统的行为
    3. 是系统完成其预期工作的能力，例如使得学生可以在线注册
    4. 可以通过使用许多可能的结构来实现
    5. 很大程度上独立于架构，因为它可以作为单个整体系统存在而没有任何内部结构

  2. Quality requirements (NFRs)（质量需求，非功能性需求）

    1. 系统应在其功能需求之上提供的整个系统的理想特性
    2. 是功能要求或整个产品的资格/质量
    3. 如果质量属性很重要，则软件体系结构会限制功能到各种结构上的分配（映射）

  3. Constraints（约束）

    1. 约束是自由度为零的设计决策，没有商量的余地，必须实现
    2. 约束是已经制定的预先指定的设计决策
    3. 通过接受设计决策并将其与其他受影响的设计决策调和，可以满足约束条件

Quality Attributes

Modeling quality attribute scenarios

质量属性的准确定义对在架构层面上进行评估是十分有必要的；质量属性场景是用来定义目标质量属性的重要方法

场景是有着确定结构的简单句子，两类主要的场景是：

  1. 通用场景是系统独立的场景，目的是去引导整个质量属性需求
  2. 具体场景是系统相关的场景，目的是去构建在质量属性在具体应用上的定义。 这是通用场景的实例。

General Scenarios

  • 通用场景提供了一个产生大量的通用，系统独立的质量属性场景的框架
  • 每个质量属性都潜在的但不是必然和我们处理的系统相关
  • 为了让通用场景对与具体的系统产生作用，我们必须把针对系统进行定制
  • 使通用情景系统变得具体意味着将其转换为特定系统的具体术语
组成元素
  • Stimulus: 当到达系统时一个需要被考虑到的条件
  • Source of Stimulus: 一个产生刺激的实体（人、系统或者任意的执行器）
  • Response: 当刺激到达后产生的活动或者回应
  • Response Measure: 对刺激的回应必须在某种程度上可以进行测量，这样对应的需求才可以被检验。
  • Environment: 当刺激发生时系统处于的条件或者环境（例如过载的、运行时等等）
  • Artifact: 整个系统或者需求对应的系统的一部分。

Tactics for quality attributes

Architecturally Significant Requirements

在架构上重要的需求是对架构产生深远影响的需求-如果没有这种需求，架构可能会显著地不同

How to gather and identify ASRs?

  1. Requirements documents

    1. 需求规范中的大部分内容都不会影响体系结构
    2. 对架构师有用的很多东西甚至在最佳需求文档中都没有

    如果需求影响关键架构设计决策，那么根据定义，它就是ASR

  2. Interviewing stakeholders

    1. 质量属性研讨会
      1. QAW展示和介绍
      2. 商业任务展示
      3. 架构计划展示
      4. 识别架构驱动程序：在精简的架构驱动程序清单上达成共识，其中包括总体要求，业务驱动程序，有害因素和质量属性
      5. 场景头脑风暴（每个利益相关者展示一个场景，表示他/她对系统的关注）
      6. 情景方案合并（整合相似场景）
      7. 通过投票确定方案优先级
      8. 场景细化：细化和阐述最重要的场景
    2. QAW的结果包括架构驱动程序列表和利益相关者（作为一个优先组）的一组QA方案

  3. Understanding business goals

  4. Utility tree

Architecture Patterns

Architecture Patterns

架构模式 是一系列 不断 在实践中使用的 设计决策 的集合，并且是对一些便于进行 重用 的特性进行描述的架构。

描述 架构模式 的 三元组： 情景，问题， 解决方案

  1. Context

    一个可以 复现 的，普遍的现实中问题发生的环境

  2. Problem

    要说明问题要求的 质量属性 ， 和遇到的 困难

  3. Solution: elements + relations + constraints

    成功解决问题的架构解决方案，可能是抽象化的

总结来说就是人们在实际的软件开发中发现的非常好用并且经常使用的设计决策和方法， 架构模式 是被 发现 的，不是被 发明 的；在新的开发情景下，总会有新的 架构模式 出现

Patterns vs. Tactics

  1. tactics 比模式简单；它们使用单一的结构或机制来应对单一的架构
  2. 模式通常将多个设计决策组合到一个包中
  3. 模式和tactics共同构成软件架构师的主要工具
  4. tactics是设计的“构建块”，可从中创建架构模式
  5. 大多数模式由几种不同的tactics组成，这些tactics可能：
    1. 都有共同的目的
    2. 经常被选择来保证不同的质量属性

例如，分层模式：

  • 增加了语义内聚
  • 限制依赖

Iteration

Reuse

Attribute-Driven Design (ADD)

8-step process:

  1. confirm requirements,
  2. choose an element to decompose
  3. identify ASRs
  4. choose a design satisfying ASRs
  5. instantiate elements & allocate responsibilities
  6. define interface
  7. verify & refine requirements
  8. repeat step 2-7 until all ASRs satisfied

输⼊: 功能需求、质量属性、约束

1. 确认有足够的需求信息

  • 系统的利益相关者已根据业务和任务目标确定了需求的优先级

  • 您可以确定在设计过程中重点关注哪些系统元素

  • 用 stimulus - response 的方法决定有没有足够的需求信息

2. 选择一个系统元素来分解

如果一开始 是 “green fileld” 所有的需求都直接被分配到整个系统

如果一开始是一个部分设计好的系统，选择系统的的一部分去继续设计（系统已被划分为各个元素，并为其分配了需求，那么选择其中一个元素重点关注）

3. 识别选择好的元素的 ASR

对所有的需求根据影响力进行排序（根据它们对体系结构的相对影响，再次对这些相同的需求进行排名，为每个需求分配“高影响”，“中影响”或“低影响”）

主要进行两个方面的排序， 一个是需求对 stake-holders 的重要性， 另一个是表明需求对体系结构的潜在影响

4. 选择满足 ASRs 的设计概念

4.1 识别设计中的问题/关键点

思考去如何去解决实际问题（如何解决设计中的ASRs），列出几种方向上的方案

4.2 列出下级关注的可选的Pattern 和 Tactics

对于每一个列出来的模式，你应该：

  • 识别每一个模式的鉴别参数，以帮助你在这些模式和tactics中进行选择
  • 估计鉴别参数的值
4.3 从列表中选择出合适的 patterns 和 tactics

思考在使用每种模式的时候会有什么不足，每种模式和其他的模式能不能合理的共存

主要考虑如下方面：

  • 使用每种模式时需要进行哪些权衡
  • 模式相互结合的程度如何
  • 是否有哪些模式是互斥的
4.4 确定 patterns/tactics 和 ASR 的关系

考虑到目前为止确定的模式和 tactics，并确定它们之间的关系；选择出的模式和 tactics 的结合很可能会产生新的pattern

4.5 描述出基本的架构视图

通过开始产生不同的架构视图来描述您选择的模式（不需要是完整的架构视图）

4.6 评估和解决这个过程中产生的不一致

查看是有否有一些没有考虑到的架构驱动因素等。

  • 根据架构驱动程序评估设计
  • 确定是否有未考虑的架构驱动程序
  • 评估可选的模式或应用其他tactics
  • 根据体系结构中其他元素的设计评估当前元素的设计，并解决所有不一致性

5. 实例化架构元素并且分配职责

产生架构分解视图并且从上到下的把架构职责进行分配。

  • 实例化您选择的每种元素的一个实例
  • 根据子元素的类型分配责任
  • 在子元素之间分配与父元素相关的责任
  • 分析并记录您所做的设计决策

6. 为实例化的元素定义接口

接口描述了PROVIDES和REQUIRES的假设，即软件元素之间相互影响

  • 行使涉及您实例化元素的功能要求
  • 观察由一个元素产生并由另一元素消费的任何信息

7. 验证和完善需求，并使其成为实例化元素的约束

确认所有的需求都已经被分配到了

  • 验证分配给父元素的所有需求是否已分配给一个或多个子元素
  • 将分配给子元素的任何职责转换为各个元素的功能需求

8. 重复上述步骤直到所有的ASRs都已经被满足

输出

  1. 软件元素：一个可以满足不同的角色和职责的计算和开发单元， 有定义好的元素， 并且和其他的软件元素相关联以组成一个系统的架构
  2. 角色： 一系列相关的职责
  3. 职责： 软件元素提供的功能、数据和信息
  4. 属性： 一个软件元素的其他信息
  5. 关系： 软件元素如何和其他元素进行关联或者交互的定义

Documenting Architecture

Views and Beyond

文档 = 视图+视图之外要额外补充的信息

Views

系统在某一个方面的信息，没有哪一个视图可以包含系统的所有信息

Styles (viewpoints), patterns and views

  1. Architectural Styles (viewpoints)

    关注组件和组件之间的交互和职责限制。更关注架构方法，不包含具体的上下文和问题

    元素的规格规范和关系类型，以及使用它们时候的约束条件； 着重于体系结构方法，并在特定风格可能有用或没有用时提供更轻量级的指导

  2. Architectural Patterns

    体现软件系统的基本结构组织；关注上下文和问题，以及如何在特定上下文环境中解决问题

    ${problem, context}–>architecture;approach$

    • 在软件工程中，架构模式是在特定情况下解决问题的通用且可重用的解决方案
    • 架构模式的目的是了解系统的主要部分如何配合在一起以及消息和数据如何通过系统流动

  3. Architectural Views

    1. 是系统中特定类型的元素及其关系的表示（并非所有系统元素，而是特定类型的那些）

    2. 视图使我们可以将系统的实体划分为感兴趣的且易于管理的系统表示形式

    3. 不同的视图支持不同的目标和用户，强调不同的系统元素和关系

    4. 不同的视图在不同程度上揭示了不同的质量属性

Structural views

  1. module views

    • Decomposition view 分解视图
    • Uses view 使用视图
    • Generalization view 概括视图
    • Layered view 分层视图
    • Aspects view 切面视图
    • Data model view 数据模型视图

    A module is an implementation unit that provides a coherent set of responsibilities.

    It is unlikely that the documentation of any software architecture can be complete without at least one module view.

    模块是一个实现单元，它提供一致职责集合

    没有至少一个模块视图，几乎不可能完成任何软件体系结构的文档编制工作

    Elements

    Modules, which are implementation units of software that provide a coherent set of responsibilities.

    模块是提供一致职责集的软件实现单元

    Relations

    • Is part of, which defines a part/whole relationship between the submodule—the part—and the aggregate module—the whole.

      Is part of：它定义了子模块（部件）和聚合模块（整体）之间的部件/整体关系

    • Depends on, which defines a dependency relationship between two modules. Specific module views elaborate what dependency is meant.

      Depends on：它定义了两个模块之间的依赖关系；特定的模块视图详细说明了依赖性的含义

    • Is a, which defines a generalization/specialization relation- ship between a more specific module—the child—and a more general module—the parent.

      Is a：它定义了更具体的模块（子模块）和更通用的模块（父模块）之间的泛化/具体化关系

    Constraints

    Different module views may impose specific topological constraints, such as limitations on the visibility between modules.

    不同的模块视图可能会施加特定的拓扑约束，例如对模块之间可见性的限制

    Usage

    • Blueprint for construction of the code

      构建代码的蓝图

    • Change-impact analysis

      变化影响分析

    • Planning incremental development

      规划增量开发

    • Requirements traceability analysis

      需求可追溯性分析

    • Communicating the functionality of a system and the structure of its code base

      交流系统的功能及其代码库的结构

    • Supporting the definition of work assignments, implementation schedules, and budget information

      支持工作分配、实现时间表和预算信息的定义

    • Showing the structure of information that the system needs to manage

      显示系统需要管理的信息结构

  2. component-and-connector views

    • Pipe-and-filter view 管道和过滤器视图
    • Client-server view 客户端-服务器视图
    • Peer-to-peer view 点对点视图
    • Service-oriented architecture view 面向服务的架构视图
    • Publish-Subscribe view 发布-订阅视图
    • shared-data view 共享数据视图
    • multi-tier view 多层视图

    Component-and-connector views show elements that have some runtime presence, such as processes, objects, clients, servers, and data stores.

    组件和连接器视图显示具有某些运行时状态的元素，例如流程，对象，客户端，服务器和数据存储

    Attachments指示哪些连接器连接到哪些组件

    Attachment是通过将连接器的端点连接到组件的端口来表现的

    Elements

    ■ Components. Principal processing units and data stores. A component has a set of ports through which it interacts with other components (via connectors).

    Components：主要处理单元和数据存储；组件具有一组端口，它通过这些端口与其他组件（通过连接器）进行交互

    ■ Connectors. Pathways of interaction between components. Connectors have a set of roles (interfaces) that indicate how components may use a connector in interactions.

    Connectors：组件之间的交互途径；连接器具有一组角色（接口），这些角色指示组件在交互中如何使用连接器

    Relations

    • Attachments. Component ports are associated with connector roles to yield a graph of components and connectors.

      Attachments：组件端口与连接器角色相关联，以生成组件和连接器图

    • Interface delegation. In some situations component ports are associated with one or more ports in an “internal” subarchitecture. The case is similar for the roles of a connector.

      Interface delegation：在某些情况下，组件端口与“内部”子体系结构中的一个或多个端口相关联；连接器的作用类似

    Constraints

    • Components can only be attached to connectors, not directly to other components.

      组件只能连接到连接器，不能直接连接到其他组件

    • Connectors can only be attached to components, not directly to other connectors.

      连接器只能连接到组件，不能直接连接到其他连接器

    • Attachments can only be made between compatible ports and roles.

      只能在兼容的端口和角色之间建立连接（Attachment）

    • Interface delegation can only be defined between two compatible ports (or two compatible roles).

      接口委托只能在两个兼容端口（或两个兼容角色）之间定义

    • Connectors cannot appear in isolation; a connector must be attached to a component

      连接器不能孤立出现； 连接器必须连接到组件

    Usage

    • Show how the system works.

      展示系统如何工作

    • Guide development by specifying structure and behavior of runtime elements.

      通过识别运行时元素的结构和行为来指导开发

    • Help reason about runtime system quality attributes, such as perfor- mance and availability

      帮助说明运行时系统质量属性，例如性能和可用性（外部质量属性是展示在系统运行时的）

  3. allocation views

    分配视图

    • Deployment view 部署视图
    • Install view 安装视图
    • Work assignment view 工作分配视图
    • other allocation views 其他分配视图

    Allocation views describe the mapping of software units to elements of an environment in which the software is developed or in which it executes.

    分配视图描述了软件单元到环境（在其中开发软件或在其中执行软件）中的元素的映射

    The usual goal of an allocation view is to compare the properties required by the software element with the properties provided by the environmental elements to determine whether the allocation will be successful or not.

    分配视图的通常目标是将软件元素所需的属性与环境元素提供的属性进行比较，以确定分配是否成功

    Allocation views can depict static or dynamic views.

    分配视图可以描述静态或动态视图

    Elements

    • Software element. A software element has properties that are required of the environment.

      软件元素具有需要从环境中获取的属性

    • Environmental element. An environmental element has properties that are provided to the software.

      环境元素具有提供给软件的属性

    Relations

    Allocated to. A software element is mapped (allocated to) an environmental element. Properties are dependent on the particular view.

    Allocated to：将软件元素映射（分配到）环境元素；属性取决于特定的视图

    Constraints

    Varies by view

    因视图而异

    Usage

    • For reasoning about performance, availability, security, and safety.

      关于性能，可用性，安全性和安全性的推理

    • For reasoning about distributed development and allocation of work to teams.

      关于分布式开发和团队工作分配的推理

    • For reasoning about concurrent access to software versions.

      用于同时访问软件版本的推理

    • For reasoning about the form and mechanisms of system installation.

      关于系统安装形式和机制的推理

Quality views

关心某个特定的质量属性，屏蔽掉系统中的其他和该质量属性无关的元素

我们可以为特定的利益相关者或解决特定的问题量身定制另一种称为质量视图的视图；这些质量视图是通过提取相关的结构视图并将它们打包在一起而形成的，例如：

  • A security view 安全视图
  • A communications view 通信视图
  • An exception or error-handling view 异常视图/错误处理视图
  • A reliability view 可靠性视图
  • A performance view 性能视图

  2. combine views

    1. 识别上表中的边际视图
    2. 通过将一个视图中的元素与另一个视图中的元素关联，将每个边际视图与具有更强选区的另一个视图结合

    例如上表中可以考虑将Decomposition和Work Assignment视图结合起来

  3. prioritize & stage

    优先排序

    1. 分解视图（这个是最重要的，决定了后续工作的任务分配和开发）

      分解视图（模块视图之一）对早期发布特别有用。 高级别（即广泛和浅层）分解通常很容易设计，并且项目经理可以利用这些信息开始为员工开发团队工作，进行培训，确定要外包的部分以及开始编制预算和进度表

    2. 80/20原则

      请注意，您不必完全满足所有涉众的所有信息需求。 提供80％的信息需要很长的路要走，这可能足够好，以便利益相关者可以完成工作。 与利益相关者联系，以查看信息的子集是否足够。 他们通常更喜欢按时交付并在预算范围内交付的产品，而不是获得完美的文档。

    3. 按顺序实现所有的视图？

      您无需先完成一个视图即可开始另一个视图。 人们可以在概述级别的信息上取得进步，采用广度优先的方法通常是最好的。

Evaluating Architecture

ATAM: Architecture Tradeoff Analysis Method

体系结构权衡分析方法

  1. Phase-0: Partnership & Preparation

    合作者与准备

    1. Stakeholders involved in ATAM

      • 评估团队领导（开发团队之外的评估团队）

      • 关键项目决策者（例如产品经理、架构师等）

    2. Inputs to and outputs of ATAM

      • Inputs: the architecture documentation

        架构文档

    3. Outputs: the evaluation plan

      评估计划

      • 谁： 利益相关者初步清单
      • 后勤（logistics）：什么时间、在哪里、怎么评估（when、where、how）
      • 何时将评估报告交付给何人
      • 评估报告应包含哪些信息/内容

  2. Phase 1: Evaluation - 1

    1. 参与者：评估团队和项目决策者

    2. Step 1~6

      1. 介绍ATAM方法(评估小组长)

        评估负责人向项目代表简要介绍ATAM，以使他们了解评估的过程和输出

      2. 介绍商业动机(项目经理或系统客户)

        项目经理或系统客户从商业角度介绍系统的概览，主要描述：

        • 其最重要的功能性需求
        • 其技术，管理，经济或政治限制
        • 其业务目标和背景
        • 其主要利益相关者
        • 架构驱动因素（构成架构的主要质量属性目标）

      3. 介绍架构(首席结构师)

        首席架构师进行演示，以适当的详细程度描述（目前为止的）架构：

        • 技术约束，例如操作系统，硬件或规定要使用的中间件
        • 系统必须与之交互的其他系统
        • 用于满足质量属性需求的体系结构方法

      4. 识别使用的架构方法(评估小组)

        • ATAM致力于通过了解架构方法来分析架构
        • 通过这一步骤，评估团队：
          • 研究了体系结构文档
          • 听过架构师的介绍
          • 询问了架构师有关系统设计中使用的模式和tactics的信息
        • 评估团队对已确定的架构方法（风格，模式和tactics）进行分类

      5. 生成质量属性效用树(评估小组和项目设计决策者) 决定性的一步

        • 这是指导其余分析的关键步骤
        • 评估团队与项目决策者一起识别和确定优先级并完善系统最重要的质量属性目标（即把最重要的质量属性找出来）
        • 质量属性目标通过质量属性效用树进行了详细说明，它通过精确定义相关的质量属性需求使需求具体化

      6. 分析架构方法(评估小组)，确保方法是正确的，获得风险点、非风险点、敏感点和权衡点列表

        • 目的是使评估团队确信，方法的实例化适合满足特定于属性的要求
        • 评估团队通过要求架构师解释架构如何相互支持来检查(来自实用程序树的)最高级别的场景（一次一个）
        • 评估团队记录相关的架构决策，并通过讨论确定并分类其风险，非风险，敏感点和权衡点
        • 该分析旨在获取足够的架构信息，以在已做出的架构决策与需要满足的质量属性之间建立某种联系
        • 在此步骤的最后，评估团队应清楚了解整个体系结构的最重要方面，关键设计决策的依据以及风险，非风险，敏感点和权衡点的清单

    3. 输出

      1. 架构简要展示
      2. 阐明业务目标
      3. 作为方案实现的特定质量属性需求的优先列表
      4. 效用树
      5. 风险和非风险点（非风险点：之前评定为局部风险，但是后来发现对整体架构产生的风险或者影响很小，这时候就会重新分配为non-risks）
      6. 敏感点和权衡点

  3. Phase 2: Evaluation - 2

    1. 参与者：评估团队，项目决策者和架构利益相关者

    2. Step (1)7～9

      1. 展示ATAM以及之前的结果

        • 重复步骤1以便新加入的利益相关者能够了解第一阶段的架构设计决策、ATAM评估方法和他们将要扮演的角色

        • 评估负责人回顾步骤2至6的结果并分享输出（效用树除外）

          效用树：先不让新加入的利益相关者知道我们第一阶段对需求优先级的排序，因为我们想知道新加入的利益相关者自己的对优先级排序的看法

      2. 头脑风暴、场景划分优先级(评估小组和利益相关者)

        • 此步骤的目的是利用更大的利益相关者社区的思考，来了解系统成功对他们而言意味着什么
        • 评估团队要求利益相关者就其各自的角色在业务上有意义的场景集思广益
        • 一旦收集了场景，就要求利益相关者对他们认为代表行为或质量问题的场景进行优先排序和合并
        • 将优先方案列表与效用树中的方案进行比较
        • 如果差异很大，则可以将其他场景识别为风险

      3. 分析架构方法

        • 在此步骤中，评估小组使用排名最高的、新生成的场景执行与步骤6相同的活动
        • 架构师解释相关的架构决策如何有助于实现每个决策

      4. 展示结果

        • 评估团队根据常见的潜在问题或系统性缺陷将风险分划为风险主题
        • 然后，将识别出的风险主题与步骤2中列出的特定业务驱动因素相关联
        • 从评估中收集的信息被汇总并提供给所有利益相关者
          • 记录的架构方法
          • 场景集及其通过头脑风暴确定的优先级
          • 效用树
          • 发现的风险和记录的非风险
          • 找到的敏感性点和权衡点
          • 风险主题和相互影响的业务驱动因素

    3. 输出

      • 从利益相关者群体获得的一个优先排序的场景列表
      • 风险主题和商业动机

  4. Phase 3: Follow-up

    跟进：对评估报告进行补充

    1. 参与者：评估团队和关键的利益相关者
    2. 输出：最终的评估报告
    3. 评估团队将编写一份书面最终报告，并分发给主要利益相关者以供审核
    4. 审核后，将报告交付给委托进行评估的人员

  5. ATAM的输出

    • 架构简要介绍
    • 阐明业务目标
    • 以质量属性场景表示的带优先级的质量属性需求
    • 效用树
    • 风险点和非风险点集合
    • 风险主题集合
    • 架构决策与质量需求之间的映射
    • 敏感点和权衡点的集合
    • 最终评估报告
