软件架构

架构师的工作：沟通、实现技术的评估、工程管理要有经验

架构来源

  • NFR：主要来源，非功能需求

  • 进一步划分：ASRs（识别出来的重要需求），质量需求（即功能性需求的实现质量）

  • stakeholder：对架构有不同的考虑

    • 用户：外部可观测的质量属性
    • 开发人员：内部质量属性（不可被观测到的）

  • organization：共同的商业目标

  • 技术环境：制约当前系统、兼容现有系统

架构视图（4+1）

四个视图描述的都是不同的方面

  • 逻辑视图：静态的结构问题

    描述组成和关系

  • 过程视图

    运行起来时各个部分相互之间的关系、行为

  • 物理视图

    软件系统和外界的某种联系，运行到特定的环境上、和外界环境的交互

  • 开发视图

    怎么开发出来的，给开发人员看的；不同的人、团队分工、协作（静态分工、开发过程、顺序、集成）

    时间、交付日期、资源

  • use case scenarios

    将四个视图结合起来；高优先级、低优先级场景

  • 设计
  • 实现
  • 维护
  • 演化

Software Architecture in General

What is software architecture?

The software architecture of a program or computing system is the structure or structures of the system, which comprise software elements, the externally visible properties of those elements, and the relationships among them. （ SEI ）

程序或计算系统的软件体系结构是系统的一个或多个结构，其中包括软件元素，这些元素的外部可见属性以及它们之间的关系。

（一个程序或者计算系统的软件架构指的是该系统的结构，它们协调了软件中的各个元素、这些元素的外部可见属性以及它们之间的关系和联系）

KeyWord: structure, element, property, relation

The fundamental organization of a system, emboied in its components, their relationships to each other and the environment, and principles governing its design and evolution. （ IEEE ）

系统的基本组织，包括其组件，彼此之间的关系以及与环境的关系，以及控制其设计和演进的原则

（ 一个系统的基本组织，体现在它的组成部分，彼此的关系和环境，以及主导其设计和演变的原则（可维护性和可扩展性））

KeyWord: organzation, compnents, relationship, principle

What does a software architect do?

软件架构师的工作 ？

  1. Liaison 协调各方（包括客户、技术团队以及业务/需求分析师），沟通交流，包括管理和市场
  2. Software Engineering 软件工程的基本技能，软件工程的最佳实践
  3. Technology Knowledge。软件开发技术的基本知识，对技术领域有深入的理解和掌握
  4. Risk Management。 风险管理能力，包括和设计、技术选择相关的风险等

Where do architectures come from?

软件架构的来源 ?

NFRs, ASRs, Quality Requirements, Stakeholders, Organisations, Technical Environments

  1. NFRs, 非功能需求，一般包括约束和质量属性
  2. Architecture Significant Requirement 架构攸关需求，即有何没有这个需求，最后设计出来的架构可能是截然不同的
  3. Quality Requirements. 在功能需求的基础上需要满足的特征
  4. Stakeholders 系统的利益相关者，例如用户、客户、开发人员等等
  5. Organisations 开发组织
  6. Techinal Environments 技术环境，包括技术选型、兼容现有系统等等

不不仅仅局限于需求，可能是商业和技术决定的集合

Architecture Views

Modern systems are frequently too complex to grasp all at once. Instead, we restrict our attention at any one moment to one (or a small number) of the software system’s structures. To communicate meaningfully about an architecture, we must make clear which structure or structures we are discussing at the moment—which view we are taking of the architecture.

The views are used to describe the system from the viewpoint of different stakeholders, such as end-users, developers, system engineer, and project managers.

现代系统常常太复杂而无法一次掌握所有信息。 取而代之的是，我们在任何时候都将注意力集中在软件系统结构中的一个（或少数）结构上。 为了对架构进行有意义的交流，我们必须弄清当前正在讨论的是哪种结构，即我们对架构的看法。

这些视图用于从不同的利益相关者（例如最终用户，开发人员，系统工程师和项目经理）的角度描述系统。

Logical view

The logical view is concerned with the functionality that the system provides to end-users.

UML diagrams used to represent the logical view include class diagrams and state diagrams。

describes architecturally significant elements of the architecture and the relationships between them.

逻辑视图与系统提供给最终用户的功能有关。

用于表示逻辑视图的UML图包括类图和状态图。

描述了架构中具有架构重要性的元素及其之间的关系；静态的结构问题，描述组成和关系

Process view

The process view deals with the dynamic aspects of the system, explains the system processes and how they communicate, and focuses on the runtime behavior of the system. The process view addresses concurrency, distribution, integrators, performance, and scalability, etc.

UML diagrams to represent process view include the activity diagram.

Concurrency and communications elements of an architecture

流程视图处理系统的动态方面，解释系统流程及其通信方式，并着重于系统的运行时行为。 流程视图解决了并发，分布，集成器，性能和可伸缩性等问题。

表示过程视图的UML图包括活动图。

架构的并发和通信元素；运行起来时各个部分相互之间的关系、行为

Physical view

The physical view depicts the system from a system engineer’s point of view. It is concerned with the topology of software components on the physical layer as well as the physical connections between these components. This view is also known as the deployment view.

UML diagrams used to represent the physical view include the deployment diagram.

how the major processes and components are mapped on to the applications hardware

物理视图从系统工程师的角度描述了系统。 它涉及物理层上软件组件的拓扑以及这些组件之间的物理连接。 该视图也称为部署视图。

用于表示物理视图的UML图包括部署图。

主要流程和组件如何映射到应用程序硬件；软件系统和外界的某种联系，运行到特定的环境上、和外界环境的交互

Development view

The development view illustrates a system from a programmer’s perspective and is concerned with software management. This view is also known as the implementation view. It uses the UML Component diagram to describe system components.

UML Diagrams used to represent the development view include the Package diagram.

*captures the internal orgnazition of software components *

开发视图从程序员的角度说明了一个系统，并且与软件管理有关。 该视图也称为实现视图。 它使用UML组件图来描述系统组件。

用于表示开发视图的UML图包括Package图。

捕获内部保存的软件组件的组织，例如配置管理工具；怎么开发出来的，给开发人员看的；不同的人、团队分工、协作（静态分工、开发过程、顺序、集成）；时间、交付日期、资源

Architecture use cases

The description of an architecture is illustrated using a small set of use cases , or scenarios, which become a fifth view. The scenarios describe sequences of interactions between objects and between processes. They are used to identify architectural elements and to illustrate and validate the architecture design. They also serve as a starting point for tests of an architecture prototype. This view is also known as the use case view.

capture the requirements for architecture, related to more than one particular view

使用一小套用例或场景来说明对体系结构的描述，这些用例或场景成为第五种视图。 这些场景描述了对象之间以及流程之间的交互序列。 它们用于标识体系结构元素以及说明和验证体系结构设计。 它们还可以作为测试体系结构原型的起点。 该视图也称为用例视图。

架构用例：捕获架构需求；与多个特定视图相关；将四个视图结合起来；高优先级、低优先级场景

注：加粗部分是老师ppt上的描述，其他是维基百科上的描述

  2. understanding the requirements 了解、理解需求
  3. Creating and selecting architecture 创建和选择架构
  4. Communicating the architecture (Stakeholders, developers) 交流架构（利益相关者，开发人员）
  5. Analysing or evaluating the architecture 分析或评估架构
    1. overall methodologies 整体方法
    2. quality specific techniques 质量特定的技术
  6. Implementing the architecture 实现架构
  7. Ensuring conformance to an architecture. 确保符合体系结构

  1. Specifying ASRs 识别ASRs
  2. Architecure design 架构设计
  3. Documenting 形成文档

Quality Attributes

Software Requirements

  1. Functional requirements（功能需求）

    1. 系统必须做什么以及系统如何给利益相关者提供价值
    2. 意味着系统的行为
    3. 是系统完成其预期工作的能力，例如使得学生可以在线注册
    4. 可以通过使用许多可能的结构来实现
    5. 很大程度上独立于架构，因为它可以作为单个整体系统存在而没有任何内部结构

  2. Quality requirements (NFRs)（质量需求，非功能性需求）

    1. 系统应在其功能需求之上提供的整个系统的理想特性
    2. 是功能要求或整个产品的资格/质量
    3. 如果质量属性很重要，则软件体系结构会限制功能到各种结构上的分配（映射）

  3. Constraints（约束）

    1. 约束是自由度为零的设计决策，没有商量的余地，必须实现
    2. 约束是已经制定的预先指定的设计决策
    3. 通过接受设计决策并将其与其他受影响的设计决策调和，可以满足约束条件

Quality Attributes

Modeling quality attribute scenarios

质量属性的准确定义对在架构层面上进行评估是十分有必要的；质量属性场景是用来定义目标质量属性的重要方法

场景是有着确定结构的简单句子，两类主要的场景是：

  1. 通用场景是系统独立的场景，目的是去引导整个质量属性需求
  2. 具体场景是系统相关的场景，目的是去构建在质量属性在具体应用上的定义。 这是通用场景的实例。

General Scenarios

  • 通用场景提供了一个产生大量的通用，系统独立的质量属性场景的框架
  • 每个质量属性都潜在的但不是必然和我们处理的系统相关
  • 为了让通用场景对与具体的系统产生作用，我们必须把针对系统进行定制
  • 使通用情景系统变得具体意味着将其转换为特定系统的具体术语
组成元素
  • Stimulus: 当到达系统时一个需要被考虑到的条件
  • Source of Stimulus: 一个产生刺激的实体（人、系统或者任意的执行器）
  • Response: 当刺激到达后产生的活动或者回应
  • Response Measure: 对刺激的回应必须在某种程度上可以进行测量，这样对应的需求才可以被检验。
  • Environment: 当刺激发生时系统处于的条件或者环境（例如过载的、运行时等等）
  • Artifact: 整个系统或者需求对应的系统的一部分。

Quality Attributes

Availability, Interoperability, Modifiability, Performance, Security, Testability, Usability, X-ability…

Tactics for quality attributes

Architecturally Significant Requirements

在架构上重要的需求是对架构产生深远影响的需求-如果没有这种需求，架构可能会显著地不同

How to gather and identify ASRs?

  1. Requirements documents

    1. 需求规范中的大部分内容都不会影响体系结构
    2. 对架构师有用的很多东西甚至在最佳需求文档中都没有

    如果需求影响关键架构设计决策，那么根据定义，它就是ASR

  2. Interviewing stakeholders

    1. 质量属性研讨会
      1. QAW展示和介绍
      2. 商业任务展示
      3. 架构计划展示
      4. 识别架构驱动程序：在精简的架构驱动程序清单上达成共识，其中包括总体要求，业务驱动程序，有害因素和质量属性
      5. 场景头脑风暴（每个利益相关者展示一个场景，表示他/她对系统的关注）
      6. 情景方案合并（整合相似场景）
      7. 通过投票确定方案优先级
      8. 场景细化：细化和阐述最重要的场景
    2. QAW的结果包括架构驱动程序列表和利益相关者（作为一个优先组）的一组QA方案

  3. Understanding business goals

  4. Utility tree
